## M-01: LimitedMinter — Reentrancy in burn() Allows Double-Spend via Callback

**Contract:** LimitedMinter.sol
**Function:** burn()
**Severity:** Medium (CVSSv3: 6.5)
**CWE:** CWE-362

### Description
The `LimitedMinter.burn()` function burns tokens from the caller and then calls `_updateBridgeBalance()` which makes an external call to the bridge contract. If the bridge contract is malicious or compromised, it can re-enter the `burn()` function before the state is updated, allowing a double-spend attack.

The function does not follow the checks-effects-interactions pattern — it burns tokens (effect) and then makes an external call (interaction) without reentrancy protection.

### Vulnerable Code
```solidity
function burn(address account, uint256 amount) external onlyRole(BURNER_ROLE) {
    _burn(account, amount);
    _updateBridgeBalance(account, amount);  // External call — reentrancy risk
}
```

### Impact
A malicious bridge contract could:
- Re-enter `burn()` multiple times before the first call completes
- Burn more tokens than the user actually approved
- Manipulate bridge balance tracking
- Cause accounting discrepancies between chains

### Recommended Mitigation
1. Apply the checks-effects-interactions pattern: move `_updateBridgeBalance()` before `_burn()`, or use a reentrancy guard.
2. Use OpenZeppelin's `ReentrancyGuard`:

```solidity
function burn(address account, uint256 amount) external nonReentrant onlyRole(BURNER_ROLE) {
    _updateBridgeBalance(account, amount);
    _burn(account, amount);
}
```

### 📝 Vulnerability Information

```
## M-02: BridgeDeposit — Fee Calculation Rounding Error Allows Small Fee Bypass

**Contract:** BridgeDeposit.sol
**Function:** _calculateFee()
**Severity:** Medium (CVSSv3: 5.3)
**CWE:** CWE-190

### Description
The `_calculateFee()` function uses integer division which truncates the result. For small amounts, the fee rounds down to zero, allowing users to bypass the fee entirely by splitting their deposits into multiple small transactions.

The fee calculation is:
```solidity
fee = (amount * feeBasisPoints) / 10000;
```

For amounts where `amount * feeBasisPoints < 10000`, the fee rounds down to 0.

### Vulnerable Code
```solidity
function _calculateFee(uint256 amount) internal view returns (uint256) {
    return (amount * feeBasisPoints) / 10000;
}
```

### Impact
- Users can avoid paying bridge fees entirely by splitting deposits
- Loss of fee revenue for the protocol
- Economic attack on the bridge's business model

### Recommended Mitigation
1. Implement a minimum fee:
```solidity
function _calculateFee(uint256 amount) internal view returns (uint256) {
    uint256 fee = (amount * feeBasisPoints) / 10000;
    return fee < minFee ? minFee : fee;
}
```

2. Or use a higher precision calculation with a scaling factor.
