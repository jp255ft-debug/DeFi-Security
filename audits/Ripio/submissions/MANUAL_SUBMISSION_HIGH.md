## H-01: BridgeDeposit — Missing Merkle Proof Verification Allows Arbitrary Minting

**Contract:** BridgeDeposit.sol
**Function:** fulfillBridgeMint()
**Severity:** High (CVSSv3: 8.5)
**CWE:** CWE-347

### Description
The `BridgeDeposit.fulfillBridgeMint()` function allows any address with `BRIDGE_OPERATOR_ROLE` to mint tokens on the destination chain without any cryptographic proof (Merkle proof or signature) that the corresponding burn actually occurred on the source chain.

The function only checks:
1. The caller has `BRIDGE_OPERATOR_ROLE`
2. The token is registered (`tokenRegistry[token] != address(0)`)
3. The transaction hash hasn't been used before (`usedTransactionHashes[txHash] == false`)

There is **no verification** that the burn event on the source chain actually happened. A malicious or compromised `BRIDGE_OPERATOR` can mint arbitrary amounts of any registered token by simply providing a fake transaction hash.

### Vulnerable Code
```solidity
function fulfillBridgeMint(
    address token,
    address to,
    uint256 amount,
    bytes32 txHash
) external onlyRole(BRIDGE_OPERATOR_ROLE) returns (bool) {
    require(tokenRegistry[token] != address(0), "Token not registered");
    require(!usedTransactionHashes[txHash], "TxHash already used");
    
    usedTransactionHashes[txHash] = true;
    IERC20Mintable(token).mint(to, amount);
    
    emit BridgeMintFulfilled(token, to, amount, txHash);
    return true;
}
```

### Impact
A malicious or compromised `BRIDGE_OPERATOR` can:
- Mint arbitrary amounts of any registered token (up to the daily limit)
- Mint tokens to any address, including themselves
- Execute multiple mints with different fake transaction hashes
- Drain the bridge's minting capacity

The only protection is the daily mint limit, which still allows significant damage before being detected.

### Recommended Mitigation
Implement a cryptographic proof verification mechanism:

1. **Merkle Proof Verification**: Store the Merkle root of all valid burn events on-chain and verify a Merkle proof against it before minting.
2. **Signature Verification**: Require a signature from a trusted off-chain oracle/relayer that witnessed the burn event.
3. **Two-Phase Commit**: Implement a commit-reveal scheme where the burn is first committed, then revealed with proof.

Example fix using Merkle proof:
```solidity
function fulfillBridgeMint(
    address token,
    address to,
    uint256 amount,
    bytes32 txHash,
    bytes32[] calldata merkleProof
) external onlyRole(BRIDGE_OPERATOR_ROLE) returns (bool) {
    require(tokenRegistry[token] != address(0), "Token not registered");
    require(!usedTransactionHashes[txHash], "TxHash already used");
    
    bytes32 leaf = keccak256(abi.encodePacked(token, to, amount, txHash));
    require(MerkleProof.verify(merkleProof, burnMerkleRoot, leaf), "Invalid Merkle proof");
    
    usedTransactionHashes[txHash] = true;
    IERC20Mintable(token).mint(to, amount);
    
    emit BridgeMintFulfilled(token, to, amount, txHash);
    return true;
}
```
